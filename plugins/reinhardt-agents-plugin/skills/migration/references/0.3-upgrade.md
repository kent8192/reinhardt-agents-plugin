# Reinhardt 0.3 Upgrade Reference

Use this reference when migrating an application from the public 0.2.x line to
0.3.0 stable. Prefer `reinhardt/instructions/MIGRATION_0.3.md` from a local
`reinhardt-web` checkout when available; this file is the bundled fallback and
triage checklist.

## Source Map

- `reinhardt/instructions/MIGRATION_0.3.md`: canonical current guide.
- Issues #5405 and #5410: release-facing migration guide and stable wording
  blockers.

## Recommended Order

1. Update `Cargo.toml` to the target 0.3 release.
2. Fix removed API references.
3. Migrate dependency providers and injection sites that need keyed identity.
4. Replace raw server-route registration with endpoint-based registration.
5. Review generated `{Model}Info` relation fields and DTO conversions.
6. Update Pages app layout and component route wrappers.
7. Regenerate migrations and review the diff.
8. Run the scan and verification commands at the end of this reference.

## Removed API Replacements

| Removed API | Replacement |
|---|---|
| `AuthUser<U>` | `CurrentUser<U>` |
| `create_resource(fetcher)` | `use_resource(fetcher, ())` |
| `create_resource_with_deps(fetcher, deps)` | `use_resource(fetcher, deps)` |
| `use_effect_event(f)` | `use_callback(closure, deps)` or `.get_untracked()` inside the effect |
| `use_effect_event_with(f, deps)` | `use_callback_with(closure, deps)` or `.get_untracked()` inside the effect |
| `ServerRouter::function`, `ServerRouter::route`, `ServerRouter::handler_with_method`, and named variants | `#[get]` / `#[post]` / endpoint macros plus `.endpoint(factory)` |
| `FunctionHandler` public re-export | endpoint-generated handler types or a custom `Handler` type registered with `.view(...)` |
| value-type-only provider identity for provider functions | `FactoryOutput<K, T>` with `Depends<K, T>` |
| `DependsResult` / `DependsOption` sugar aliases | `Depends<K, Result<T, E>>` / `Depends<K, Option<T>>` |

## Auth Extractor

Use `CurrentUser<U>` as the full authenticated-user extractor.

```rust
// Before
use reinhardt_auth::AuthUser;

async fn profile(AuthUser(user): AuthUser<MyUser>) -> Response {
    let id = user.id();
    // ...
}

// After
use reinhardt_auth::CurrentUser;

async fn profile(CurrentUser(user): CurrentUser<MyUser>) -> Response {
    let id = user.id();
    // ...
}
```

Cookie-backed session apps should register `SessionMiddleware` once in
`urls.rs`. Standard `SessionData` + `SessionAuthExt` + `CurrentUser<U>` setups
do not need an additional session-auth middleware layer.

## Keyed Dependency Providers

Use direct `Injectable` values when the type itself is the unique dependency
identity. Use `FactoryOutput<K, T>` when a provider function returns a value type
that may have multiple meanings in one application.
In these examples, `DbSettings` is an application-owned settings type or
fragment that the app registers as an injectable value.

```rust
// Before: provider identity was the value type.
#[injectable_factory(scope = "singleton")]
async fn database(
    #[inject] settings: DbSettings,
) -> DatabaseConnection {
    DatabaseConnection::connect(&settings.database_url).await.unwrap()
}
```

```rust
// After: provider identity is the explicit key.
use reinhardt::di::{Depends, FactoryOutput, injectable, injectable_key};

#[injectable_key]
struct PrimaryDatabase;

#[injectable(scope = "singleton")]
async fn database(
    #[inject] settings: DbSettings,
) -> FactoryOutput<PrimaryDatabase, DatabaseConnection> {
    FactoryOutput::new(
        DatabaseConnection::connect(&settings.database_url)
            .await
            .unwrap(),
    )
}

async fn health(
    #[inject] db: Depends<PrimaryDatabase, DatabaseConnection>,
) -> Response {
    Response::ok()
}
```

For fallible initialization, keep the key first and put `Result<T, E>` in the
provider value position:

```rust
#[injectable(scope = "singleton")]
async fn database(
    #[inject] settings: DbSettings,
) -> FactoryOutput<PrimaryDatabase, Result<DatabaseConnection, DbError>> {
    FactoryOutput::new(DatabaseConnection::connect(&settings.database_url).await)
}

async fn health(
    #[inject] db: Depends<PrimaryDatabase, Result<DatabaseConnection, DbError>>,
) -> Response {
    Response::ok()
}
```

`#[injectable_factory]` is a deprecated compatibility alias for provider
functions. New code should use `#[injectable]`.

## Server Route Registration

Move raw function registrations to endpoint macros and register endpoint
factories with `.endpoint(...)`.

```rust
// Before
async fn health(_request: Request) -> Result<Response> {
    Ok(Response::ok())
}

let router = ServerRouter::new()
    .function("/health", Method::GET, health);
```

```rust
// After
#[get("/health", name = "health")]
async fn health() -> ViewResult<Response> {
    Ok(Response::ok())
}

let router = ServerRouter::new()
    .endpoint(health);
```

Keep `.view(...)` / `.view_named(...)` for class-style handlers that
intentionally implement generic `Handler` behavior. Prefer endpoint macros for
HTTP-method endpoints.

## Resources and Effects

Use `use_resource(fetcher, deps)` for both mount-only and dependency-driven
resource loading:

```rust
let questions = use_resource(fetch_questions, (page,));
let user = use_resource(fetch_user, ());
```

Replace `use_effect_event` with callbacks when the effect should call a stable
function, or read the latest signal value with `.get_untracked()` when that
value should not become an effect dependency.

## Pages Layout

0.3 route wrappers should live under `src/apps/<app>/client/components/`, not
in a separate `pages.rs` wrapper.

```rust
use reinhardt::pages::component;
use reinhardt::pages::component::Page;

use crate::apps::polls::client::components::nav::with_nav;

#[component("/polls/{question_id}/", name = "detail")]
pub fn polls_detail(question_id: Path<i64>) -> Page {
    with_nav(super::polls_detail(question_id.into_inner()))
}
```

> **0.4.0-alpha.1+ route-name migration:** route-backed components require a
> named string-literal argument. Rewrite `#[component(PATH, "name")]` as
> `#[component(PATH, name = "name")]`. For
> `#[component(PATH, route_name)]`, choose the intended public route name and
> write it explicitly as `name = "..."`; identifier shorthand is rejected.

Generated Pages apps use app-local split modules:

```text
src/apps/polls.rs
src/apps/polls/
├── client.rs
├── client/
│   ├── components.rs
│   └── components/
├── server.rs
├── server/
│   ├── admin.rs
│   ├── forms.rs
│   ├── models.rs
│   └── views.rs
├── serializers.rs
├── server_fn.rs
├── services.rs
├── services/client.rs
├── services/server.rs
├── urls.rs
└── urls/
    ├── client_router.rs
    └── server_router.rs
```

Migration actions:

- Delete app-local `pages.rs` and `client/pages.rs` wrappers after moving route
  wrappers into `client/components/*.rs`.
- Rename `urls/server_urls.rs` to `urls/server_router.rs`.
- Keep `urls/client_router.rs` behind `#[cfg(client)]` and
  `urls/server_router.rs` behind `#[cfg(server)]`.
- Move server-only `admin`, `forms`, `models`, and `views` under the app-local
  `server/` directory.
- Split target-specific services into `services/client.rs` and
  `services/server.rs`.
- Preserve generated empty directories with `.gitkeep` when templates expect
  users to add submodules later.
- Remove broad project-level `shared/forms.rs` and `shared/types.rs` files
  unless the application still has hand-written cross-app DTOs there.

## WASM/Native Parity

Application code under `src/apps/**`, `src/config/**`, and shared DTO modules
should compile without broad call-site `#[cfg]` workarounds. Relevant parity
changes:

- `#[user]` is inert on WASM, so shared user model declarations can stay visible
  to browser builds while native builds get auth runtime behavior.
- `#[client_page]` emits native route-table stubs so client routes can be named
  from shared route aggregates.
- `#[injectable]` emits inert WASM provider stubs and skips native-only
  registration side effects.

## Model Info, Partial Updates, and Migrations

Generated `{Model}Info` relation fields now expose relation-shaped payloads:

- one-to-one / foreign-key fields use `RelationInfo<T>`,
- many-to-many fields use `ManyToManyInfo<Source, Target>`.

Review API DTOs, serializer outputs, and browser tests that previously expected
flattened `*_id` scalar fields.

`QuerySet` supports atomic conditional partial updates with
`.update_fields([...])`. Empty assignment lists and predicate-less partial
updates are rejected at the API boundary.

Regenerate and review migrations after:

- model relations changed and generated info types changed shape,
- field renames are now unambiguous enough to emit `RenameColumn`,
- replayed migrations previously produced spurious `AlterColumn` or
  `AddConstraint` operations.

## Scan Commands

Run these scans before editing and again before final verification:

```bash
SCAN_DIRS=()
for dir in src crates examples; do
  [[ -e "$dir" ]] && SCAN_DIRS+=("$dir")
done

PAGE_SCAN_DIRS=()
for dir in src examples; do
  [[ -e "$dir" ]] && PAGE_SCAN_DIRS+=("$dir")
done

if ((${#SCAN_DIRS[@]})); then
  rg -n "AuthUser|create_resource|create_resource_with_deps|use_effect_event|use_effect_event_with" "${SCAN_DIRS[@]}"
  rg -n "ServerRouter::route(_named)?\\(|\\.(function|handler_with_method)(_named)?\\(|FunctionHandler|Depends(Result|Option)" "${SCAN_DIRS[@]}"
  rg -n "FactoryOutput<|Depends<[^,>]+>|injectable_factory|InjectableKey" "${SCAN_DIRS[@]}"
  rg -n "\\.route(_named)?\\(" "${SCAN_DIRS[@]}" -g '*server_router.rs' -g '*server_urls.rs'
fi

if ((${#PAGE_SCAN_DIRS[@]})); then
  rg --files "${PAGE_SCAN_DIRS[@]}" | rg '(^|/)(pages\.rs|server_urls\.rs|client/pages(/|\.rs$)|src/shared/(forms|types)\.rs$)'
fi
```

## Verification

Run the narrow checks first, then the relevant project gates:

```bash
cargo check --workspace --all-features
cargo test --doc
```

For apps with broader coverage, also run their normal unit, integration, Pages,
database, and WASM checks.
